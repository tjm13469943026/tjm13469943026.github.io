iOS中UITableView的详细使用

2016-11-23
技术

##scorllView（滚动视图）的总结：
几个常用属性：

1.scrollView的contentSize是用户能滚动的整体大小，而scrollView的frame中的size是其窗口的大小，既用户每次能看见的大小。如果向scroll中添加图片，建议吧contentSize设置成跟图片大小一样。

2.contentInset是内边距地设置，弹簧效果的关键词是bounces（bool变量），滚动条的关键词是indicator。

3.缩放：这个需要代理来实现UIScrollViewDelegate，viewForZoomingInScrollView方法是返回一个UIViwe，以确认哪个视图需要放大缩小，常用Tag值来标记。放大系数的关键词是zoomscale。
还有一个比较常用搞得方法是scrollViewDidScroll（滚动视图发生滚动的时候）可以监滚动视图的偏移量contentOffset.x和contentOffset.y，两个不同方向的偏移量。

##tableView的总结：

###1.重用机制：

当创建出tableView后，系统会生成一个NSMutableArray visibaleCells和一个NSMutableDictnery reusableTableCell，假定有100个cell（它们的类型一致）需要显示，而屏幕最大能显示10个，那么实际上只生成了11份cell的内存，而且这11个cell都有同一个重影标识符，当第11个cell完全显示出来的时候，第一个cell消失在屏幕中，cell1加入到reusaDictnery中，键是重用标识符，值是cell的内存首地址，由于只有一种类型的cell，所以可知整个字典最多只有一组键值对，如果tableView中有多种类型的cell，我们需要给不同类的cell设置不同的标识符，这样的话，字典中就会有多个键值对。cell11同时也会加入到visibaleCells的数组中，当继续往下滑，cell12会从重用字典中寻找跟他具有相同重用标识符的cell，由于字典中的cell1跟他具有同样的键，所以cell12会直接在cell1的基础上修改一下数据显示在屏幕上，并不会重新创建一份内存，同时cell2从屏幕中移除，cell2进入重用字典，由于cell1跟cell2有相同的键，所以cell1会从重用字典中移除，当然如果cell1跟cell2的键不一样，既他们有不同的重用标识符，则cell1不会被移除。

###2.关于每一组中cell如何显示（tableView的重点难点）
首先有4种方法

（1）老式方法

首先是从重用字典中寻找是否有可重用的cell，然后用if进行判断，如果没有则自行创建cell，cell的风格，以及内部子视图可以根据需要在if中创建。代码示例如下：

UITableViewCell *cell = [tableView dequeueReusableCellWithIdentifier:identify];

cell = [[UITableViewCell alloc] initWithStyle:UITableViewCellStyleDefault reuseIdentifier:identify];

//if内写样式(Cell长什么样)
cell.accessoryType = UITableViewCellAccessoryDisclosureIndicator;

UIImageView *imageView = [[UIImageView alloc]initWithFrame:CGRectZero];

//规定:cell上的内容都必须加到cell的contentView中
[cell.contentView addSubview:imageView];

[imageView mas_makeConstraints:^(MASConstraintMaker *make) {
make.width.height.equalTo(40);
make.centerY.equalTo(0);
make.right.equalTo(-20);
}];
imageView.tag = 1000;
}

//对于内容的设置,要写在if外

cell.textLabel.text = @”账号管理”;

UIImageView imageView = (UIImageView )[cell.contentView viewWithTag:1000];
imageView.image = [UIImage imageNamed:@”aio_voiceChange_effect_2”];

(2）老式方法

首先是从重用字典中寻找是否有可重用的cell，然后用if进行判断，如果没有通过，则通过xib创建，代码示例如下：

UITableViewCell *cell = [tableView dequeueReusableCellWithIdentifier:identify];
cell = [[NSBundle mainBundle] loadNibNamed:@”DaRenCell” owner:nil options:nil].firstObject;

如果需要更改cell中数据，则需要在xib文件中给每个子视图设置好tag值，然后根据tag值找到对应的地址进行修改。

（3）新式方法

首先需要创建一个tableViewCell的类，用于管理cell具体显示内容，代码示例如下：

-(instancetype)initWithStyle:(UITableViewCellStyle)style reuseIdentifier:(NSString *)reuseIdentifier{
self = [super initWithStyle:UITableViewCellStyleValue1 reuseIdentifier:reuseIdentifier];
if (self) {
self.accessoryType = UITableViewCellAccessoryDisclosureIndicator;
}
return self;
}

可以看到，是重写初始化方法来达到自定义cell的目的，当然你也可以不创建类，那么在使用新式方法的时候，系统创建cell的风格是默认风格。
然后是需要在合适的位置注册该cell，代码示例如下：

[_tableView registerClass:[RightDetailCell class] forCellReuseIdentifier:@”RDCell”];

下面是使用该cell：

RightDetailCell *cell = [tableView dequeueReusableCellWithIdentifier:@”RDCell” forIndexPath:indexPath];

cell.textLabel.text = indexPath.row ? @”消息记录” : @”消息通知”;

具体的过程叙述如下：
首先是去该tableView的重用字典中寻找是否有@“RDCell”的键，如果没有，则系统会自动创建cell，而创建cell是根据注册的信息来的，此处我们是让系统去用我们自定义的类来创建，而且创建的时候会把注册时候给的重用标识符传递到初始化方法中（可以看到初始化方法中带有一个reusableIdentifier的参数），所以创建出来的cell都被打上该重用标识符的烙印，以便下次使用，而cell得具体样式，会根据初始化方法中的代码进行塑造，如果是比较复杂的cell，那么需要在自定义类中声明属性，以便在使用该cell的时候根据需要修改。

（4）新式方法

同样需要创建一个类，同时也创建一个跟该类同名的xib文件，注册时候需要注意，是通过xib文件注册了，代码如下：

[_tableView registerNib:[UINib nibWithNibName:@”LockCell” bundle:nil] forCellReuseIdentifier:@”LockCell”];

cell的具体样式是根据xib中的布局来决定的，需要修改的子视图可以连线到该类的头文件中，以便在调用的时候更改数据。具体的过程跟新式方法1一样。

###3.Cell的编辑模式

（1）删除与添加
首先是需要让cell在编辑模式下能显示出能够删除跟添加的按钮，代码示例如下：
//规定某一行的编辑模式

-(UITableViewCellEditingStyle)tableView:(UITableView )tableView editingStyleForRowAtIndexPath:(NSIndexPath )indexPath{
if (indexPath.row == self.dataList.count - 1) {

//最后一行被设置成添加模式

return UITableViewCellEditingStyleInsert;
}

//其他被设置删除模式

return UITableViewCellEditingStyleDelete;
}

这个方法是设置具体某一行能进入编辑模式：
//指定某一行是否可以编辑

-(BOOL)tableView:(UITableView )tableView canEditRowAtIndexPath:(NSIndexPath )indexPath{
//取巧: 0 是NO, 其他数字都是YES
return indexPath.row;
}

下面这个方法是设置tableView进入编辑模式：
[_tableView setEditing:!_tableView.isEditing animated:YES];可以添加一个segmentedControl来控制编辑模式。

下面这个方法是点击删除、添加按钮以后的数据跟UI操作：

-(void)tableView:(UITableView )tableView commitEditingStyle:(UITableViewCellEditingStyle)editingStyle forRowAtIndexPath:(NSIndexPath )indexPath{
switch (editingStyle) {
case UITableViewCellEditingStyleNone: {

break;
}
case UITableViewCellEditingStyleDelete: {
//删除操作触发
//因为界面是根据数据来显示内容的, 所以删除时, 必须先删除数据, 再更新界面.
[self.dataList removeObjectAtIndex:indexPath.row];
[tableView deleteRowsAtIndexPaths:@[indexPath] withRowAnimation:UITableViewRowAnimationAutomatic];
break;
}

case UITableViewCellEditingStyleInsert: {
[self.dataList addObject:@”达内教育”];

//最后一行的位置

NSIndexPath *ip = [NSIndexPath indexPathForRow:self.dataList.count - 1 inSection:0];

//第二参数的效果, 自己下课测试

[tableView insertRowsAtIndexPaths:@[ip] withRowAnimation:UITableViewRowAnimationAutomatic];
break;
}
}
}

整个添加删除的过程到此才结束。

#总结一下：
首先通过按钮等控制整个tableView能够进入编辑模式，有一个方法可以规定具体哪些cell可以进入编辑模式，canEditRowAtIndexPath
。
进入编辑模式后，cell上具体显示的是什么编辑状态，需要用到editingStyleForRowAtIndexPath方法，值得注意的是删除按钮跟添加按钮不能同时出现。
最后是对编辑按钮进行实际的改写，否则只是有按钮，并不会出现删除跟添加的实际操作。这时候需要用到commitEditingStyle方法，具体的删除跟添加遵循的原则是操作数据，然后操作UI。

（2）移动操作
同样是需要一个返回bool的方法判断哪行可以出现移动图标：canMoveRowAtIndexPath这个方法是单独控制某一行是否出现移动图标，当然canEditRowAtIndexPath这个方法是比它权利更大。
然后是具体的移动了，然后是具体的操作了，它单独是一个方法，跟添加删除是独立的，代码示例如下：

  • (void)tableView:(UITableView )tableView moveRowAtIndexPath:(NSIndexPath )sourceIndexPath toIndexPath:(NSIndexPath *)destinationIndexPath{
    //移动UI的同时, 对数据项进行同步的修改
    //1.获取正在移动的数据
    id obj = self.dataList[sourceIndexPath.row];
    //2.把数据从原来位置删除
    [self.dataList removeObject:obj];
    //3.把数据插入到新的位置
    [self.dataList insertObject:obj atIndex:destinationIndexPath.row];
    }
    由代码可以看到我们只是操作了数据，并没有操作UI。

##4.关于表头表尾、组头组尾
表头表尾是整个tableView的，而且只有一个表头一个表尾，而且表头表尾不能共用一个UIView，要分开写。
而组头组尾相对来说就要复杂一些，因为tableView可能会有很多个section，这个就涉及到重用机制了，需要创建一个tableView组头组尾专属的类来管理，并在类的初始化方法中根据需要改写，同时在使用之前也是要注册的，也涉及到重用标识符，跟cell的一模一样。值得一提的是如果组头组尾是UIView，如果需要接收点击事件，可以根据需要添加一个UIControl在视图中。
